# Лабораторная работа №5 — Технический долг проекта Money Flow

## Цель работы

Цель лабораторной работы — провести инспекцию учебного проекта Money Flow на наличие технического долга по каждому из признаков, разработать и оценить план мероприятий по его устранению, а также сопоставить объём долга с объёмом нереализованных функций и сделать обоснованные выводы.

## Теоретическая часть

### Понятие технического долга

Технический долг (Technical Debt) — это совокупность архитектурных, кодовых и процессных проблем, которые возникают при принятии краткосрочных решений в ущерб долгосрочному качеству системы. Подобно финансовому долгу, технический долг «начисляет проценты»: чем дольше он остаётся неустранённым, тем выше стоимость его погашения.

Важно различать намеренный и ненамеренный долг. Намеренный — это осознанный выбор: «сейчас быстро, потом перепишем». Ненамеренный возникает из-за недостаточной квалификации или изменившихся требований. В учебных проектах преобладает второй тип.

### Признаки технического долга

 Согласно методическим материалам к лабораторной работе, выделяют следующие признаки технического долга:

- Непонятный / нечитабельный код
- Дублирующийся код
- Отсутствие автоматизации (тестов, сборки, развёртывания)
- Запутанная архитектура и ненужные сложные зависимости
- Медленные / неэффективные средства разработки
- Незакоммиченный код / долгоживущие ветки
- Отсутствие или несоответствие технической документации
- Отсутствие тестовой среды
- Длинные циклы интеграции / отсутствие непрерывной интеграции

## Описание проекта

Money Flow — мобильное приложение для учёта личных финансов, разработанное на Flutter/Dart с использованием SQLite (sqflite) в качестве локального хранилища данных.

На момент проведения инспекции в проекте реализованы следующие эпики и пользовательские истории:

- **Эпик 1 (Кошельки):** Создание, просмотр, редактирование, удаление кошельков; переключение активного кошелька.
- **Эпик 2 (Операции):** Добавление, редактирование, удаление доходов, расходов и переводов между кошельками с автоматическим пересчётом баланса.
- **Эпик 3 (Категории):** Создание, редактирование, удаление категорий; привязка категорий к операциям доходов и расходов.

Используемые технологии: Flutter, Dart, sqflite (SQLite), Material Design 3, `setState` для управления состоянием.

## Инспекция технического долга

### 1. Непонятный / нечитабельный код

**Статус: Частично**

Основная проблема сосредоточена в классе `DatabaseHelper`. На текущий момент этот файл содержит более 300 строк кода и включает CRUD-методы сразу для трёх сущностей: `Wallet`, `TransactionModel` и `Category`. Дополнительно там же реализованы специфические операции: `createTransfer`, `deleteTransaction` с разветвлённой логикой по типу операции, `setActiveWallet`.

Метод `deleteTransaction` особенно сложен для понимания: он содержит вложенную транзакцию с несколькими ветками (`income`, `expense`, `transfer`), каждая из которых выполняет разные SQL-запросы и обновляет балансы разных кошельков. Отсутствуют комментарии к этим веткам.

Также в методе определения парной операции перевода используется хрупкая эвристика на основе анализа строкового значения поля `note` (проверка, содержит ли строка подстроку `'в "'` или `'на "'`), что делает код трудным для сопровождения.

```dart
// Хрупкая эвристика — признак технического долга
final isOut = note.contains('в "') || note.contains('на "');
```

**Рекомендация:** разбить `DatabaseHelper` на три отдельных DAO-класса, добавить XML-doc комментарии к публичным методам, заменить строковую эвристику на поле `transfer_group_id` в таблице.

### 2. Дублирующийся код

**Статус: Есть**

Логика обновления баланса кошелька дублируется как минимум в четырёх местах:

| Метод | Паттерн |
|---|---|
| `createTransaction` | читать баланс → прибавить/вычесть → записать |
| `updateTransaction` | читать баланс → откатить старое → применить новое → записать |
| `deleteTransaction` | читать баланс → откатить → записать |
| `createTransfer` | читать балансы двух кошельков → изменить оба → записать |

Каждая реализация самостоятельно выполняет `txn.query('wallets', ...)`, проверяет наличие записи, извлекает `balance` и затем делает `txn.update`. Это прямое нарушение принципа DRY (Don't Repeat Yourself).

**Рекомендация:** выделить приватный вспомогательный метод `_adjustWalletBalance(txn, walletId, delta)`, который принимает знаковое значение `delta` и атомарно изменяет баланс. Все четыре метода выше начнут использовать его.

```dart
// Целевое состояние
Future<void> _adjustWalletBalance(
  DatabaseExecutor txn, int walletId, double delta) async {
    final result = await txn.query('wallets',
        where: 'id = ?', whereArgs: [walletId], limit: 1);
    final current = (result.first['balance'] as num).toDouble();
    await txn.update('wallets', {'balance': current + delta},
        where: 'id = ?', whereArgs: [walletId]);
}
```

### 3. Отсутствие автоматизации (тесты, CI/CD)

**Статус: Есть — наиболее критичный долг**

На момент инспекции в проекте:

- Нет ни одного unit-теста (файла `*_test.dart`).
- Нет widget-тестов.
- Нет integration-тестов.
- Нет настроенного CI/CD (GitHub Actions или аналога).

Это означает, что каждое изменение в `DatabaseHelper` проверяется исключительно вручную. На практике именно отсутствие тестов уже привело к реальному багу: при удалении перевода откатывался баланс только одного кошелька из двух. Баг был обнаружен и исправлен вручную, но не защищён регрессионным тестом.

Критичные сценарии, которые необходимо покрыть тестами:

- Создание транзакции типа `income` корректно увеличивает баланс кошелька.
- Создание транзакции типа `expense` корректно уменьшает баланс кошелька.
- `createTransfer` атомарно изменяет балансы обоих кошельков.
- `deleteTransaction` для `transfer` откатывает балансы обоих кошельков и удаляет обе связанные записи.
- Невозможно создать перевод, если сумма превышает баланс источника.
- Удаление кошелька с операциями не нарушает целостность БД.

**Рекомендация:** добавить зависимость `flutter_test` (уже входит в SDK), написать тесты с использованием in-memory SQLite (`sqflite_ffi`), настроить GitHub Actions workflow для автозапуска тестов при пуше.

### 4. Запутанная архитектура и лишние зависимости

**Статус: Частично**

В отчёте по лабораторной работе №3 задекларирована трёхслойная архитектура: Presentation → Data → Models. На практике этот принцип частично нарушен:

- Все экраны (`HomeScreen`, `WalletListScreen`, `CategoriesScreen`) напрямую обращаются к `DatabaseHelper.instance` — глобальному синглтону. Это нарушает принцип Dependency Inversion: UI-компоненты зависят от конкретной реализации хранилища, а не от абстракции.
- Весь state management реализован через `setState` + метод `_loadData()` в каждом экране. При каждом изменении данных экран перезапрашивает все данные целиком (включая повторные SQL-запросы для сумм за месяц, последних транзакций и т.д.), что неэффективно.
- `HomeScreen` выполняет прямые `rawQuery` к базе данных, то есть содержит SQL-логику в слое представления.

```dart
// В HomeScreen — SQL в UI слое, признак нарушения архитектуры
final incomeResult = await database.rawQuery(
  '''SELECT SUM(amount) AS total FROM transactions
     WHERE wallet_id = ? AND type = 'income' AND date >= ?''',
  [active.id, monthStart.toIso8601String()],
);
```

**Рекомендация:** ввести слой репозиториев (`WalletRepository`, `TransactionRepository`) с интерфейсами; перенести все SQL-запросы в DAO-классы; перейти на `Provider` или `Riverpod` для управления состоянием.

### 8. Отсутствие тестовой среды

**Статус: Есть**

В проекте нет отдельной тестовой базы данных. Все ручные проверки выполняются на «боевых» данных в эмуляторе. Это означает, что при тестировании деструктивных операций (удаление кошелька, удаление перевода) разработчик вынужден вручную воссоздавать тестовые данные.

Отсутствие mock-объектов или in-memory БД также не позволяет писать изолированные unit-тесты для `DatabaseHelper` без поднятия реального SQLite.

**Рекомендация:** использовать `sqflite_ffi` для тестов с in-memory базой; создать `TestDataFactory` — класс-помощник для быстрого заполнения тестовыми данными (кошельки, операции, категории).

***

### 9. Длинные циклы интеграции / отсутствие CI

**Статус: Есть**

Сборка и проверка кода выполняются только локально, вручную. Нет автоматического запуска `flutter analyze` или `flutter test` при коммите. Это приводит к тому, что ошибки анализа или регрессии могут оставаться незамеченными до момента ручного запуска приложения.

**Рекомендация:** добавить `.github/workflows/flutter.yml` с шагами: `flutter pub get` → `flutter analyze` → `flutter test`.

## Сравнение: технический долг vs нереализованные фичи

### Объём технического долга

Общий объём выявленного технического долга оценивается примерно в **31 час**.

### Объём нереализованных фич

По плану бэклога проекта нереализованными остаются два эпика:

- **Эпик 4 (Управление аккаунтом):** регистрация, вход, редактирование профиля, синхронизация данных между устройствами — оценочно **30–40 часов**.
- **Эпик 5 (Статистика и аналитика):** графики и отчёты по доходам/расходам за период, визуализация по категориям — оценочно **20–30 часов**.

Итого нереализованных фич: **50–70 часов**.

### Анализ соотношения

| Категория | Оценка (ч) | Доля |
|---|---|---|
| Технический долг | ~31 | ~35% |
| Нереализованные фичи (Эпики 4–5) | ~60 (среднее) | ~65% |
| **Всего** | **~91** | **100%** |

Соотношение типично для учебного проекта на стадии активного прототипирования: функциональность развивается быстрее, чем качество кода. Объём долга (~31 ч) составляет примерно треть от суммарных оставшихся трудозатрат — это пороговое значение, при котором откладывание рефакторинга начинает существенно замедлять дальнейшую разработку.

Критически важно, что часть долга (особенно отсутствие тестов и строковая эвристика в `deleteTransaction`) уже проявилась в виде реального бага: при удалении перевода откатывался баланс только одного кошелька. Это означает, что долг не просто «потенциальная проблема», а уже влияет на корректность работы приложения.

## Выводы и корректировки плана разработки

### Обоснование необходимости устранения долга

Устранение долга **частично необходимо** — в первую очередь в тех частях, где он уже приводит к ошибкам или будет блокировать реализацию Эпика 4. Полная переработка архитектуры до начала новых фич не оправдана по соотношению затрат и выгод.

**Устранить немедленно (Фаза 1):**
- Дублирование логики баланса — риск внесения новых ошибок при дальнейшем изменении расчётов.
- Строковая эвристика в `deleteTransaction` — хрупкое решение, которое сломается при изменении текста примечания.

**Устранить в Спринте 3 (Фаза 2):**
- Написать тесты на критичные сценарии — без них каждая новая фича несёт риск регрессии.
- Настроить CI — минимальные трудозатраты при максимальном долгосрочном эффекте.

**Совместить с Эпиком 4 (Фаза 3):**
- Рефакторинг архитектуры (репозитории, Provider) логично провести при добавлении синхронизации, так как без нормального state management работа с несколькими устройствами будет практически невозможна.

### Скорректированный план разработки

| Спринт | Содержание | Приоритет |
|---|---|---|
| Спринт 3 | Фаза 1 (быстрые победы) + Фаза 2 (тесты, CI, разбивка DAO) | Рефакторинг |
| Спринт 4 | Эпик 4 (аккаунт) + Фаза 3 (репозитории, Provider) | Фичи + архитектура |
| Спринт 5 | Эпик 5 (статистика, графики) | Фичи |

Такой порядок обеспечивает: устранение наиболее критичного долга до реализации сложного Эпика 4, постепенное повышение качества кодовой базы без остановки разработки, и защиту от регрессий через тесты уже с третьего спринта.
